# Конспект: Поведенческие паттерны проектирования (GoF)

---

## 1. Посетитель (Visitor)

**Суть одним словом:** Разделение данных и операций над ними.

**Проблема:** Есть стабильный набор классов, но постоянно добавляются новые
операции над ними. Если класть операции внутрь классов — они раздуваются.

**Решение:** Выносим все операции в отдельные классы-«посетители».
Каждый элемент данных вызывает `accept(visitor)`, а visitor через
двойную диспетчеризацию попадает в нужный метод `visitXxx()`.

**Когда применять:**
- Иерархия классов стабильна (редко добавляются новые типы)
- Часто добавляются новые, несвязанные операции (осмотр, экспорт, ремонт…)
- Нужно отделить «что это за объект» от «что с ним можно сделать»

**Плюсы:** +OCP (новые операции без правки классов), +SRP (классы хранят только данные)
**Минусы:** −Нарушает инкапсуляцию, −Сложно если часто добавляются новые типы элементов

**Пример:** `SellVisitor` и `UseVisitor` для лута в Escape From Stash.

## 2. Стратегия (Strategy)

**Суть одним словом:** Набор взаимозаменяемых алгоритмов, выбираемых на лету.

**Проблема:** Один и тот же результат можно получить разными алгоритмами
(сортировка, скидки, шифрование). `if/else` внутри класса — спагетти.

**Решение:** Каждый алгоритм — отдельный класс с общим интерфейсом.
Контекст хранит ссылку на стратегию и делегирует ей работу.

**Когда применять:**
- Есть несколько вариантов одного алгоритма
- Нужно переключать поведение во время выполнения
- Хочется избавиться от ветвления `if (type == A) … else if (type == B) …`

**Плюсы:** +Лёгкое переключение, +Тестирование каждого алгоритма отдельно, +OCP
**Минусы:** −Лишние классы, −Клиент должен знать, какую стратегию выбрать

**Пример:** `PriceCalculator` со стратегиями `NoDiscount`, `PercentOff`, `FlatOff`.

---

## 3. Состояние (State)

**Суть одним словом:** Объект меняет поведение при смене внутреннего состояния.

**Проблема:** Много ветвлений `if (state == LOCKED) … else if (state == OPEN) …`
в каждом методе. Добавление нового состояния — правка всего класса.

**Решение:** Каждое состояние — отдельный класс с общим интерфейсом.
Контекст делегирует поведение текущему состоянию. Состояние само решает,
на какое переключиться.

**Когда применять:**
- Поведение объекта зависит от его состояния в рантайме
- Громоздкие `switch`/`if-else` по состояниям
- Нужно легко добавлять новые состояния и переходы

**Плюсы:** +SRP (каждое состояние — свой класс), +Нет спагетти-условий, +Явные переходы
**Минусы:** −Много классов, −Контекст не знает о соседних состояниях

**Пример:** Турникет метро: `Locked` (монета → Unlocked) ↔ `Unlocked` (толчок → Locked).

## 4. Наблюдатель (Observer)

**Суть одним словом:** Один меняется — многие об этом узнают.

**Проблема:** Один объект меняет состояние, и об этом должны узнать несколько
других. Жёсткие ссылки на всех — высокая связанность.

**Решение:** Подписчики (`Observers`) регистрируются у издателя (`Subject`).
При изменении Subject вызывает `notify()`, и каждый Observer реагирует.

**Когда применять:**
- Одно событие должно уведомить несколько независимых компонентов
- Подписчики динамические (добавляются/удаляются во время работы)
- Нужна слабая связанность: издатель не знает, кто подписан

**Плюсы:** +Слабая связанность, +Динамическая подписка, +Можно добавлять подписчиков не трогая издателя
**Минусы:** −Неизвестен порядок уведомлений, −Могут быть утечки памяти если не отписаться

**Пример:** `NewsPublisher` → подписчики `EmailService`, `SmsService`, `PushService`.

## 5. Команда (Command)

**Суть одним словом:** Действие = объект (можно сохранить, отложить, отменить).

**Проблема:** Вызов метода привязан к объекту-получателю. Невозможно
отменить действие, поставить в очередь или записать в лог.

**Решение:** Каждое действие — объект с методом `execute()` и `undo()`.
Объект содержит получателя и параметры. Отправитель (Invoker)
просто вызывает `execute()` и хранит историю.

**Когда применять:**
- Нужна отмена/повтор (undo/redo)
- Действия нужно ставить в очередь или логировать
- Макрокоманды (несколько действий как одно)

**Плюсы:** +Undo/RedO из коробки, +Очереди и логирование, +Разделение отправителя и получателя
**Минусы:** −Много классов-команд, −Клиент должен создавать объект команды

**Пример:** Корзина: `AddItemCommand`, `RemoveItemCommand` с историей для отмены.

## 6. Цепочка обязанностей (Chain of Responsibility)

**Суть одним словом:** Запрос идёт по цепочке, пока кто-то не обработает.

**Проблема:** Неизвестно, кто именно обработает запрос. Жёсткий `if-else`
по всем возможным обработчикам — сложно поддерживать.

**Решение:** Каждый обработчик хранит ссылку на следующий. Получив запрос,
он либо обрабатывает его, либо передаёт дальше.

**Когда применять:**
- Несколько объектов могут обработать запрос, но неизвестно кто именно
- Обработчик определяется динамически
- Нужна возможность менять порядок или состав цепочки

**Плюсы:** +Отправитель не знает, кто обработает, +Гибкая цепочка, +SRP
**Минусы:** −Запрос может остаться без обработки, −Сложно отлаживать

**Пример:** Техподдержка: Бот → Уровень 1 → R&D (каждый пытается обработать или передать дальше).

## 7. Шаблонный метод (Template Method)

**Суть одним словом:** Скелет алгоритма в базовом классе, шаги — в потомках.

**Проблема:** Несколько классов делают почти одно и то же, но с мелкими
отличиями в шагах. Код дублируется, порядок шагов может нарушиться.

**Решение:** Базовый класс задаёт жёсткий порядок шагов (`final` метод),
а конкретные шаги — абстрактные или хуки, которые переопределяют потомки.

**Когда применять:**
- Алгоритмы имеют одинаковый порядок, но отличаются реализацией шагов
- Нужно гарантировать, что шаги всегда выполняются в одном порядке
- Есть опциональные шаги (hooks), которые можно переопределить

**Плюсы:** +Защита от изменения порядка, +Устранение дублирования, +Хуки для расширения
**Минусы:** −Наследование (жёсткая связь), −Нельзя поменять порядок шагов в рантайме

**Пример:** `SyncJob`: `lock → fetch → transform → persist → unlock` (fetch/persist у подклассов свои).

## 8. Итератор (Iterator)

**Суть одним словом:** Обход коллекции без знания её внутреннего устройства.

**Проблема:** Разные коллекции хранят данные по-разному (массив, список,
дерево). Клиент не должен знать эти детали.

**Решение:** Выносим логику обхода в отдельный объект-итератор
с методами `hasNext()` и `next()`.

**Когда применять:**
- Нужен последовательный доступ без раскрытия внутреннего устройства
- Несколько одновременных обходов одной коллекции (независимые курсоры)
- Ленивая загрузка или кастомный порядок обхода

**Плюсы:** +SRP (обход отдельно от хранения), +Независимые курсоры, +Единый интерфейс
**Минусы:** −Лишний объект, −Для простых массивов избыточен

**Пример:** `Playlist` (связный список) с итератором, позволяющим два независимых обхода.

## 9. Посредник (Mediator)

**Суть одним словом:** Все общаются через одного посредника, а не напрямую.

**Проблема:** Много объектов общаются друг с другом напрямую →
паутина зависимностей. Изменение одного ломает десяток других.

**Решение:** Все компоненты знают только о посреднике. Посредник
знает обо всех и маршрутизирует сообщения.

**Когда применять:**
- Сложная сеть взаимодействий (многие-ко-многим)
- Логика координации часто меняется
- Нужен централизованный контроль и валидация

**Плюсы:** +Слабая связанность, +Централизация логики, +Легче тестировать
**Минусы:** −Посредник может стать «божественным объектом» (God Object)

**Пример:** `SignupMediator` координирует `UserController`, `UserRepository`, `Mailer`.

---

## 10. Хранитель (Memento)

**Суть одним словом:** Снимок состояния для последующего восстановления.

**Проблема:** Нужно сохранять и восстанавливать состояние объекта,
но не нарушать инкапсуляцию (не делать все поля публичными).

**Решение:** Объект создаёт «снимок» (Memento) через метод `createMemento()`.
Хранитель (Caretaker) хранит снимки. Восстановление через `restore(memento)`.
Внутренние данные снимка недоступны никому, кроме исходного объекта.

**Когда применять:**
- Нужна история состояний (undo/redo, автосохранение)
- Нельзя нарушать инкапсуляцию (прямой доступ к полям)
- Снимки создаются нечасто (экономия памяти)

**Плюсы:** +Сохранение инкапсуляции, +Простое undo/redo, +История
**Минусы:** −Память (каждый снимок — копия), −Сложно при большом состоянии

**Пример:** Текстовый редактор: `save()` → `Memento(text, cursor)`, `restore(memento)`.

## Сводная таблица

| Паттерн | Решает проблему… | Ключевая идея |
|---------|-------------------|---------------|
| **Visitor** | Новые операции для стабильных классов | Операции = отдельные классы |
| **Strategy** | Выбор алгоритма в рантайме | Алгоритмы = взаимозаменяемые объекты |
| **State** | Поведение зависит от состояния | Состояния = полиморфные классы |
| **Observer** | Уведомление многих об изменении одного | Подписка + автоматическое оповещение |
| **Command** | Отмена/очередь/логирование действий | Действие = объект с execute/undo |
| **Chain of Resp.** | Неизвестный обработчик запроса | Запрос идёт по цепочке до первого обработчика |
| **Template Method** | Дублирование алгоритма с вариациями | Скелет в базе, шаги у потомков |
| **Iterator** | Обход без знания внутреннего устройства | Курсор = отдельный объект |
| **Mediator** | Паутина зависимостей между объектами | Все общаются через одного посредника |
| **Memento** | Сохранение/восстановление без нарушения инкапсуляции | Снимок состояния = непрозрачный объект |
